# 《ta说》Trae Vibe-Coding 指引

## 1. 使用目标

本文档用于指导后续在 `Trae` 中进行 `vibe-coding`，目标不是让 AI 一次性“生成整个项目”，而是让它按清晰模块、稳定边界、可验收任务逐步交付。

适用对象：

- 开发者 A：恐怖游戏部分
- 开发者 B：翻译官与分享卡部分

核心原则：

1. 一次只让 `Trae` 做一个完整而有限的任务。
2. 先要求结构和边界，再要求视觉和细节。
3. 所有 prompt 都要写清楚输入、输出、限制、验收标准。
4. 让 `Trae` 优先修改指定文件，避免大范围失控改动。
5. 生成后必须人工 review，不直接相信状态机、数据结构和本地存储逻辑。

---

## 2. 通用协作原则

### 2.1 推荐的提问结构

每次发给 `Trae` 的需求，尽量包含以下 6 个元素：

1. 背景：这个模块在《ta说》里是做什么的。
2. 范围：只改哪些文件，只实现哪些功能。
3. 限制：不要动哪些模块，不要引入什么复杂库。
4. 数据：明确输入输出类型或状态字段。
5. 交付：希望它产出代码、文档、组件还是 mock。
6. 验收：你会怎么判断它完成了。

### 2.2 最有效的任务粒度

适合：

- 新建一个页面
- 新建一个 service
- 实现一个状态模块
- 补一个 API 封装
- 修一个明确 bug
- 为现有组件补交互

不适合：

- “把整个项目全部做完”
- “把所有页面都美化一下”
- “你自己决定整体架构并实现”

### 2.3 每次生成前必须告诉 `Trae` 的约束

固定可复用约束：

```text
你是这个项目的协作工程师。请严格基于现有文件结构和共享类型工作。
不要重命名已有核心类型，不要改动未提及模块，不要自行引入复杂状态管理库。
先保证功能闭环和可读性，再做视觉优化。
如果存在多种方案，优先选择最简单、最稳定、最容易联调的实现。
```

## 3. 给开发者 A 的指引

### 3.1 开发者 A 适合让 `Trae` 做的事

- 搭聊天 UI 骨架
- 实现发送消息逻辑
- 实现关键词污染规则模块
- 实现草稿监听逻辑
- 实现读档状态管理
- 实现真相页逐行播放
- 实现空间页阶段切换

### 3.2 开发者 A 不建议直接让 `Trae` 决定的事

- 核心剧情节奏
- 所有污染文案的最终版本
- 读档是否回滚哪些字段
- 多个 Meta 事件的优先级冲突

这些必须由你先给规则，再让它编码。

### 3.3 开发者 A 的标准 prompt 模板

```text
我在做《ta说》，这是一个恋爱 Meta 恐怖互动网页项目。

现在请你只完成【聊天主界面消息发送链路】这一项，不要改其他页面。

技术栈：
- React + TypeScript
- Zustand 管理会话状态

请你在以下约束下实现：
1. 用户可以输入文字并点击发送。
2. 发送后先把原文写入聊天流。
3. 第 3 次发送时，必须触发第一次反义污染。
4. 污染逻辑先用本地规则，不接真实接口。
5. 不要实现复杂视觉动效，只保留一个简单的“发送中...”状态。
6. 不要改动翻译官相关页面。

请优先修改这些文件：
- src/pages/ChatPage/*
- src/features/pollution/*
- src/types/*

如果缺少类型，请补最小必要类型。

完成标准：
- 聊天页可以发送消息
- 第 3 次发送必触发污染
- 污染前后文本都能在状态里看到
- 代码结构清晰，可继续接 TA 回复系统
```

### 3.4 开发者 A 的分任务 prompt

#### Prompt A1：聊天 UI 骨架

```text
请为《ta说》实现一个手机竖屏聊天页骨架，风格接近极简聊天 App。

只做这些：
1. 顶部栏：返回、昵称、状态、空间/读档/退出按钮
2. 中间消息流：支持用户消息、TA 消息、系统消息三种气泡
3. 底部输入栏：文本输入框、语音按钮占位、发送按钮

要求：
1. 用 React + TypeScript 组件拆分。
2. 不要接真实业务逻辑，用 mock 数据。
3. 样式保持克制，不要过度设计。
4. 组件命名清晰，方便后续接状态管理。
```

#### Prompt A2：关键词污染引擎

```text
请为《ta说》实现一个 `pollutionEngine`，只处理本地规则，不接接口。

输入：
- userInput: string
- stage: StoryStage
- fearType: FearType

输出：
- originalText
- pollutedText
- triggerReason
- keyword
- shouldShowBeforeAfter

要求：
1. 至少内置 8 个关键词污染规则。
2. 第 3 次发送必须可被外部强制触发污染。
3. 如果没命中关键词但需要污染，请返回一个按 fearType 分类的 fallback 结果。
4. 保持纯函数，便于测试。
```

#### Prompt A3：草稿监听

```text
请实现《ta说》的输入框草稿监听模块 `draftMonitor`。

业务规则：
1. 监听输入框文本变化。
2. 如果文本中出现敏感词，标记为候选草稿。
3. 如果用户删除或清空，把被删除片段记录到 `deletedDrafts`。
4. 需要避免记录太短、太碎的噪声内容。

要求：
1. 输出为可复用 hook 或工具模块。
2. 提供最小使用示例。
3. 不要直接耦合页面 UI。
```

#### Prompt A4：读档系统

```text
请实现《ta说》的 `saveLoadManager`。

规则：
1. 在第一次污染前创建一个自动存档。
2. 第 1 次读档：恢复聊天快照，但保留 loadCount。
3. 第 2 次读档：恢复时插入异常消息。
4. 第 3 次读档：不恢复，直接返回失控结果。
5. metaMemory 和 deletedDrafts 不回滚。

请输出：
1. TypeScript 类型
2. manager 实现
3. 一组简单的用例或调用示例
```

#### Prompt A5：真相页

```text
请实现《ta说》的真相揭示页组件。

要求：
1. 页面全黑背景，文字逐行出现。
2. 支持按数组传入文案，每行有延迟。
3. 播放结束后触发 `onComplete`。
4. 组件不要耦合具体剧情文案，文案由外部传入。
```

### 3.5 开发者 A 的 review 清单

每次让 `Trae` 改完后，你重点检查：

1. 是否偷偷改了共享类型名字。
2. 是否把剧情控制写死在 UI 组件里。
3. 是否把 localStorage 读写散落到多个文件。
4. 是否出现难以测试的大组件。
5. 是否破坏“规则优先，LLM 次之”的原则。

## 4. 给开发者 B 的指引

### 4.1 开发者 B 适合让 `Trae` 做的事

- 搭翻译官页面
- 实现结构化结果卡片
- 封装翻译 API client
- 实现分享卡页面
- 组装分享卡数据结构
- 补截图导出能力

### 4.2 开发者 B 不建议直接让 `Trae` 决定的事

- 恋爱翻译官最终输出风格边界
- 分享卡的传播文案方向
- 哪些会话字段代表结局类型

这些需要你先定规则，再让它实现。

### 4.3 开发者 B 的标准 prompt 模板

```text
我在做《ta说》，这是一个前半段是恋爱 Meta 恐怖游戏、后半段是恋爱翻译官的项目。

现在请你只完成【翻译官结果展示模块】。

技术栈：
- React + TypeScript

约束：
1. 只改翻译官相关页面和组件。
2. 不要修改聊天页和状态机。
3. 结果必须结构化展示以下 5 个字段：
   - 原话
   - 可能真实含义
   - 尖锐翻译
   - 更好的表达
   - 实操建议
4. 先用 mock 数据驱动。
5. 样式方向是高级、克制、可截图，不要花哨。

完成标准：
- 页面支持空态、加载态、结果态、错误态
- 结构清晰，后续能无痛接接口
```

### 4.4 开发者 B 的分任务 prompt

#### Prompt B1：翻译官页面

```text
请为《ta说》实现 `TranslatorPage` 页面。

要求：
1. 页面包含多行输入框、提交按钮、结果展示区。
2. 结果区支持空态、loading、error、success 四种状态。
3. success 状态按结构化字段展示。
4. 先使用本地 mock service，不接真实接口。
5. 页面应适合移动端竖屏展示。
```

#### Prompt B2：翻译服务封装

```text
请实现《ta说》的 `translatorService`。

要求：
1. 暴露一个 `translateLoveLanguage(chatText)` 方法。
2. 当前先提供 mock 实现，同时保留未来切真实 API 的接口层。
3. 返回类型固定为 `LoveTranslationReport`。
4. 如果输入为空或过短，请返回可供 UI 展示的错误。
```

#### Prompt B3：分享卡数据生成器

```text
请实现《ta说》的 `shareCardComposer`。

输入：
- 会话统计数据
- 翻译官结果

输出：
- endingType
- hardestSentence
- shareLine
- pollutionCount
- deletedDraftCount
- loadCount
- fearType
- aiTranslation

要求：
1. 保持纯函数。
2. 如果缺少 shareLine，给出本地 fallback。
3. 不要在函数内部直接访问 localStorage。
```

#### Prompt B4：分享卡页面

```text
请实现《ta说》的分享卡页面。

要求：
1. 页面视觉像“关系幻觉报告”。
2. 显示结局名、最大嘴硬句、AI 翻译、统计数据、最后一句话。
3. 提供一个导出按钮，先预留导出方法，不一定接完整库。
4. 保持样式克制，黑白灰为主，少量红色提示。
```

#### Prompt B5：首页通关后入口

```text
请为《ta说》的开始页增加“进入恋爱翻译官”按钮。

规则：
1. 只有 `hasFinishedGame === true` 时才显示。
2. 保留“开始聊天”作为主按钮。
3. 不要改动聊天主流程。
```

### 4.5 开发者 B 的 review 清单

1. 结果字段名是否与共享类型一致。
2. UI 是否直接耦合某个 mock 数据结构。
3. 分享卡是否写死了某些统计值。
4. 错误态和 loading 态是否完整。
5. 是否方便未来切换本地模板和真实 API。

## 5. 联调时怎么问 `Trae`

### 5.1 适合集成期的 prompt

```text
请帮助我联调《ta说》的通关后跳转逻辑。

当前情况：
- 聊天主流程在通关后会写入 hasFinishedGame
- 翻译官页面已经存在
- 首页需要根据本地状态显示“进入恋爱翻译官”

请你只检查并修改这些内容：
1. 通关写入 hasFinishedGame 的逻辑
2. 首页读取并展示入口按钮
3. 梦醒页跳转到翻译官页

不要改聊天 UI，不要改分享卡，不要引入新库。
请给出最小必要修改。
```

### 5.2 适合修 bug 的 prompt

```text
请帮我修一个明确问题，不要重构无关代码。

问题描述：
第 2 次读档后，loadCount 被错误重置为 0，导致第 3 次读档没有进入失控剧情。

请你：
1. 先定位 bug 原因
2. 只修改相关状态恢复逻辑
3. 如果需要，补一个最小测试或示例
4. 不要改页面样式和其他功能
```

## 6. 推荐工作流

### 6.1 单个任务的最佳流程

1. 先自己定义边界和验收。
2. 把需求缩到一个模块。
3. 发给 `Trae` 生成第一版。
4. 人工 review 结构和类型。
5. 让 `Trae` 做第二轮修正。
6. 本地跑通后再进入下一个模块。

### 6.2 不要这样用

不要直接说：

- “根据 PRD 把整个项目全做了”
- “你自己安排架构并开始编码”
- “帮我把所有内容补齐”

这样会让 `Trae` 容易：

1. 擅自改目录
2. 擅自发明类型
3. 混淆游戏前半段和翻译官后半段
4. 产出无法联调的大段代码

## 7. 推荐 prompt 套餐

### 套餐 A：从零搭建项目

按顺序问：

1. 生成目录结构和共享类型
2. 生成聊天页骨架
3. 生成翻译官页骨架
4. 生成污染引擎
5. 生成翻译服务
6. 生成分享卡页面

### 套餐 B：冲 Demo

按顺序问：

1. 先做最小聊天主链路
2. 再做第一次污染
3. 再做真相页
4. 再做翻译官 mock
5. 最后做分享卡静态页

### 套餐 C：补完整度

按顺序问：

1. 草稿监听
2. 读档系统
3. 空间异常页
4. 首页通关后入口
5. 导出分享卡
6. 接真实 API

## 8. 交付验收模板

你可以把下面这段直接复制给 `Trae`：

```text
在你完成后，请按以下格式自检并告诉我：

1. 你改了哪些文件
2. 每个文件新增了什么职责
3. 是否改动了共享类型
4. 当前功能如何手动验证
5. 有哪些已知限制或后续待接内容

如果你需要做超出范围的改动，请先停下来说明原因，不要自行扩展范围。
```

---

## 9. 最终建议

最适合《ta说》的 `Trae` 使用方式不是“全自动做项目”，而是：

1. 你们先定清楚状态机、页面边界和共享类型。
2. 再让 `Trae` 逐块生成 UI、service、engine。
3. 恐怖感、节奏感、传播感这些高层判断由人把控。
4. `Trae` 主要负责把你们的规则快速落成可运行代码。

这样效率最高，也最不容易把项目做歪。
