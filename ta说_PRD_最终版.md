# 《过拟合恋人》PRD 最终版

## 0. 文档信息

- 项目名称：`过拟合恋人`
- 项目类型：恋爱 Meta 恐怖轻互动网页游戏 + 恋爱语言翻译工具
- 当前开发定位：正式开发版
- 核心关键词：恋爱聊天、反义污染、Meta 恐怖、读档记忆、输入框监听、恋爱语言翻译、关系幻觉报告

## 1. 项目一句话

《过拟合恋人》不是一个普通恋爱助手，而是一部先让用户亲身体验“误解如何发生”，再把这些误解翻译成清晰关系语言的互动作品。

前半段，用户进入一段逐渐失控的恋爱聊天。  
后半段，系统把这段体验回收到真实沟通问题中，告诉用户“你真正想说的到底是什么”。

## 2. 产品目标

1. 做出一条完整、正式、可长期维护的互动产品链路，而不是演示性质原型。
2. 让用户在 5-8 分钟内完成一轮从聊天、失控、梦醒到翻译官和分享卡的闭环体验。
3. 让规则系统负责剧情推进和触发逻辑，让模型负责受约束的文本细节生成。
4. 让恋爱翻译官成为通关后可真实使用的正式功能，而不是静态展示页或假数据结果。
5. 让分享卡成为正式传播载体，基于真实会话数据与真实生成结果输出。

## 3. 非目标

当前不做：

1. 多章节长篇剧情。
2. 多角色、多地图、多路线恋爱养成。
3. 头像上传、通讯录读取、相册读取、真实聊天软件读取。
4. 真实定位调用。
5. 心理诊断、医学建议、关系裁决。
6. 鼓励操控、冷暴力、PUA、报复性沟通等输出。

注意：  
“不做”是产品范围约束，不是允许用简化替身功能代替正式功能的理由。

## 4. 正式开发原则

1. 不以 `MVP`、`Demo`、`mock 闭环` 作为验收口径。
2. 不使用静态页、假接口、预设结果、固定短句去代替正式功能。
3. 不把“以后再接 LLM”“以后再做语音”写成当前文档的默认实施前提。
4. 若某项正式能力暂时不可用，应明确标记为未完成，而不是以代偿实现冒充完成。
5. 所有文档、README、协作提示词和实现要求，统一以本 PRD 为准。

## 5. 用户与使用场景

### 5.1 目标用户

1. 对恋爱、暧昧、聊天潜台词感兴趣的年轻用户。
2. 喜欢 Meta 叙事、轻恐怖、聊天模拟器、互动小说的用户。
3. 愿意截图分享体验结果的社交传播用户。
4. 对 AI 互动作品和语言翻译能力感兴趣的用户。

### 5.2 核心使用场景

#### 场景 A：完整互动体验

用户进入网页，完成关系恐惧与 `TA` 称呼选择，进入聊天界面。  
随着互动推进，用户会经历：

- 发言被反义污染；
- 删掉的话被 `TA` 点破；
- 读档后 `TA` 依然保留记忆；
- 空间页与退出按钮参与叙事；
- 最终发现 `TA` 不是普通恋人，而是由反话、误解、草稿和投射组成的存在。

#### 场景 B：通关后翻译真实聊天

用户通关后进入恋爱翻译官，输入一段真实或模拟聊天记录，系统输出：

- 原话
- 可能真实含义
- 尖锐翻译
- 更好的表达
- 实操建议

#### 场景 C：分享传播

用户生成分享卡并导出图片。  
分享卡上展示：

- 结局类型
- 最大嘴硬句
- 系统替你发出的真话
- AI 翻译结果
- 反义污染次数
- 删除草稿次数
- 读档失败次数
- 一句总结短句

## 8. 页面与交互要求

### 8.1 开始页

目标：

- 让用户知道这是恋爱聊天互动作品
- 不提前剧透完整恐怖机制
- 明确说明输入仅在当前页面中用于互动体验

必须包含：

- 项目名 `过拟合恋人`
- 副标题
- 开始按钮
- 通关后“进入恋爱翻译官”入口

### 8.2 选择页

用户必须选择两个维度：

1. 关系恐惧类型：
   - 害怕被抛下
   - 害怕被控制
   - 害怕说真话
2. `TA` 称呼：
   - 他
   - 她
   - TA

### 8.3 聊天主界面

页面结构：

- 顶部栏：返回、昵称、状态、空间 / 读档 / 退出
- 中间区域：聊天气泡流、系统提示、故障文本、异常消息
- 底部栏：文本输入框、语音按钮、发送按钮

状态变化必须可见：

- 在线
- 正在输入...
- 正在读取...
- 已同步
- 记忆恢复中
- 你无法退出
- `TA` 不在线，但仍在回复

### 8.4 真相页与梦醒页

真相页要求：

- 页面变黑
- 聊天气泡消失
- 文字逐行出现

梦醒页要求：

- 手机界面恢复正常
- 聊天记录回到昨晚普通对话
- 显式提供进入恋爱翻译官的入口

### 8.5 恋爱翻译官页

必须支持：

1. 文本输入
2. 语音输入转文字
3. 输出结构化结果
4. 结果进入分享卡

不得用以下代替正式能力：

- 固定假结果
- 静态示例卡片
- 无模型参与的硬编码整页输出

### 8.6 分享卡页

必须支持：

1. 基于真实会话数据与真实翻译结果展示
2. 生成关系幻觉报告
3. 导出图片

不得以以下方式视为完成：

- 静态写死页面
- 只展示占位字段
- 不依赖真实会话状态

## 9. 核心机制一：反义污染

### 9.1 定义

用户输入一句话后，系统在实际发送前，将其改写成：

- 相反意思；
- 更暴露潜台词的意思；
- 更容易引发误解的意思；
- 更像“你不敢说但系统替你说了”的意思。

重点不是逐词取反，而是反转情绪、立场和关系意图。

### 9.2 触发方式

采用三种正式触发：

1. 剧情节点触发：第 3 次发送后，必定触发第一次污染。
2. 关键词触发：命中特定关键词时触发。
3. 反义时间段触发：中段进入 30 秒污染期。

### 9.3 关键词表

至少支持以下关键词：

- 没事
- 随便
- 忙
- 想你
- 喜欢
- 太快
- 害怕
- 分开
- 不要
- 以后再说
- 算了
- 都可以
- 晚安
- 对不起

### 9.4 发送动画

流程必须包含：

用户原文  
→ 点击发送  
→ 短暂显示原文  
→ 显示“发送中...”  
→ 文字闪烁 / 乱码 / 抖动  
→ 替换为污染后文本  
→ `TA` 回复

### 9.5 Before / After

第一次污染必须明确展示 `Before / After`。  
后续展示频率可以降低，但用户必须能清楚感知“我原本说的”和“实际发出去的”不同。

## 10. 核心机制二：恐怖回复系统

### 10.1 设计原则

`TA` 的回复不应总是解释，而应形成四级强度：

1. 轻微不对劲
2. 明显诡异
3. 压迫和非人感
4. 真相前疯癫

允许使用：

- 极短句
- 句号
- 重复
- 停顿
- 呵呵
- 读取行为痕迹的语气

### 10.2 回复来源原则

`TA` 回复必须以真实生成链路完成。  
不得以“长期固定回复库直接顶替正式回复能力”作为产品交付。

规则系统负责提供上下文和强约束；
模型负责在约束内输出 1-2 句真实聊天风格回复。

## 11. 核心机制三：输入框监听

系统监听用户输入的内容。  
若用户输入过敏感词后又删除，需要记录该草稿片段，并在中后期被 `TA` 点破。

必须监听的敏感词至少包括：

- 害怕
- 想你
- 喜欢
- 分开
- 没事
- 算了
- 不要
- 对不起
- 太快
- 救命

页面必须明确提示：

本作品仅在当前页面内处理输入内容，用于剧情互动。

## 12. 核心机制四：读档 / 存档系统

读档的目标不是复杂游戏存档，而是制造 Meta 恐怖感。

必须只设置一个自动存档点，并实现三种读档状态：

| `loadCount` | 按钮文案 | 点击效果 |
|---|---|---|
| `0` | 读档 | 恢复至自动存档点 |
| `1` | 读档失败 | 恢复但插入异常消息 |
| `2+` | TA 已读取上个存档 | 不恢复，触发失控剧情 |

读档时必须满足：

- `chatHistory` 可回滚到快照
- `loadCount` 不回滚
- `metaMemory` 不回滚
- `deletedDrafts` 不回滚

## 13. 核心机制五：语音输入

语音输入是正式能力，不是可有可无的代偿项。

正式要求：

1. 用户点击语音按钮后启动语音识别
2. 语音识别结果写回输入框
3. 用户确认后进入同样的污染与发送流程

如果当前运行环境不支持语音能力，应明确提示“当前环境不支持语音输入”，而不是用预设文本选项去伪装成语音功能完成。

## 14. 核心机制六：空间异常页

空间页不是可有可无的附属页，而是正式叙事证据链的一部分。

它必须展示从“正常甜蜜动态”到“监控式观察”的变化。

至少包含：

1. 正常牵手图与温柔配文
2. 监控风格异常图与 `I SEE YOU`
3. 标题从 `TA 的空间` 逐渐变化到 `你的空间`

注意：  
此能力不需要真实用户照片或真实社交数据，使用项目设计素材即可。这是产品边界，不是功能代偿。

## 15. 核心机制七：退出异常

退出按钮必须成为 Meta 恐怖元素。

流程至少包含：

1. 第一次点击：弹窗但不真正退出
2. 第二次点击：按钮文案变化
3. 后期点击：`TA` 明确参与回复并阻止退出的心理完成

## 16. 核心机制八：定位惊吓

定位惊吓是叙事素材，不调用真实定位。

实现要求：

- 使用虚构定位图素材
- 图中表现两个点几乎重合
- 强化“TA 一直就在这里”的惊吓感

这属于隐私边界设计，不属于功能代偿。

## 17. 真相揭示与主结局

真相必须明确揭示：

- 用户没有在和真实恋人聊天
- 后来的 `TA` 是由未发送草稿、反话、误解和期待生成的存在
- 它擅长把用户的反话翻译成怪物

梦醒后：

- 聊天恢复为普通朋友 / 暧昧对象的现实状态
- 首页与梦醒页都必须开放恋爱翻译官入口

## 18. 恋爱翻译官

### 18.1 功能定义

恋爱翻译官是正式能力。  
它将用户输入的聊天记录翻译成更清楚、更可回应的表达。

### 18.2 入口规则

- 首次体验：必须通关后出现
- 通关后：首页增加“进入恋爱翻译官”按钮
- 通过 `localStorage` 记录 `hasFinishedGame`

### 18.3 输入类型

正式支持：

1. 单句聊天
2. 多轮聊天记录
3. 自己想回复但不知道怎么说的话
4. 语音输入转文字

### 18.4 输出结构

必须固定为：

- 原话
- 可能真实含义
- 尖锐翻译
- 更好的表达
- 实操建议

### 18.5 风格约束

可以：

- 犀利
- 直接
- 轻微扎心
- 指出表达问题
- 给具体建议

不可以：

- 羞辱用户
- 断言对方一定不爱你
- 给心理诊断
- 鼓励操控、冷暴力、跟踪、PUA、报复
- 替用户做重大情感决策

### 18.6 正式能力要求

以下做法都不能算“翻译官已完成”：

- 静态展示假结果
- 固定示例文本反复输出
- 只做 UI，不提供真实翻译链路

## 19. 分享卡设计

### 19.1 定位

分享卡是用户的“关系幻觉报告”，不是普通结果页。

它必须满足：

1. 有截图欲望
2. 有传播点
3. 有扎心短句
4. 有游戏数据
5. 有 AI 翻译结果
6. 视觉上像异常聊天诊断单

### 19.2 标题与结构

主标题：

`《过拟合恋人》关系幻觉报告`

分享卡必须包含：

- 结局类型
- 最大嘴硬句
- 系统替你发出的真话
- AI 翻译
- 污染次数
- 删除草稿次数
- 读档失败次数
- 关系幻觉类型
- 最后一行总结短句

### 19.3 结局类型

正式结局命名采用：

- 草稿幽灵
- 反话感染者
- 循环恋人
- 未发送的告白
- 亲密逃亡者
- 过度理解者

### 19.4 导出

分享卡必须可导出为图片。

## 20. LLM 设计

### 20.1 使用原则

核心原则：

1. 剧情推进靠规则。
2. 文本细节靠 LLM。
3. LLM 不决定剧情走向。
4. 失败提示应明确，不以假结果代替正式结果。

### 20.2 功能分工

| 内容 | 控制方式 |
|---|---|
| 剧情阶段 | 前端规则控制 |
| 关键词触发 | 前端规则控制 |
| 恐怖事件 | 前端规则控制 |
| 反义表达 | 规则优先 + LLM 补充 |
| TA 回复 | 规则约束下的 LLM 生成 |
| 恋爱翻译官 | LLM 结构化生成 |
| 分享卡总结 | 模板 + LLM 一句话 |

### 20.3 Prompt 需求

至少需要四类正式 prompt：

1. 反义污染生成
2. `TA` 回复生成
3. 恋爱翻译官
4. 分享卡一句话生成

这些 prompt 必须输出稳定、可解析、可验证的结果。

### 21.3 隐私原则

1. 不做头像上传
2. 不调用真实定位
3. 不读取通讯录
4. 不读取真实聊天软件

## 22. 技术方案要求

### 22.1 前端

推荐：

- `Vite + React`
- 移动端 / 聊天界面适配
- `localStorage` 管理存档和通关状态
- CSS 或轻量动画实现故障效果

### 22.2 接口

正式接口建议包括：

- `POST /api/anti-translate`
- `POST /api/ta-reply`
- `POST /api/love-translate`
- `POST /api/share-line`

### 22.3 正式交付要求

以下不视为正式交付：

- 本地 mock API 长期替代正式能力
- 预设输入替代真实语音输入
- 预设回复替代正式 `TA` 生成链路
- 静态分享卡替代真实数据与导出

## 24. 验收标准

### 24.1 功能验收

必须满足：

1. 用户可以从开始页进入游戏
2. 用户可以选择关系恐惧与 `TA` 称呼
3. 用户可以输入文字
4. 用户可以使用真实语音输入
5. 至少 8 个关键词可触发反义污染
6. 至少 1 次输入框监听成立
7. 读档按钮至少 3 个状态
8. 空间页至少有正常图和异常图
9. 用户可以进入主结局
10. 用户可以进入恋爱翻译官
11. 恋爱翻译官输出结构化分析
12. 用户可以看到并导出分享卡
13. 通关后首页可以直接进入恋爱翻译官

### 24.2 体验验收

必须满足：

1. 5-8 分钟内完成一轮体验
2. 1 分钟内出现第一次强互动
3. 3 分钟内出现第一次 Meta 恐怖事件
4. 用户能明确感知 `Before / After`
5. 用户能理解“反义污染”机制
6. 用户能感受到 `TA` 的异常
7. 结尾能理解作品真正讨论的问题
8. 分享卡有传播欲

### 24.3 交付验收

以下情况视为未完成：

1. 关键功能仅以静态页代替
2. 关键生成功能仅以假数据代替
3. 语音输入仅以预设文本代替
4. 翻译官仅做结构展示不做正式生成
5. 分享卡不依赖真实会话状态

## 25. 评审对齐

### 25.1 玩法与交互设计

- 聊天输入即交互
- 语音输入增强参与感
- 消息被反义污染
- 输入框被监听
- 读档按钮会变化
- 空间按钮会异常
- 退出按钮参与叙事

### 25.2 创意与惊喜感

- 恋爱聊天突然变成 Meta 恐怖
- `TA` 读取草稿
- `TA` 记得读档
- 空间页从正常动态变成监控异常
- 结尾反转为恋爱翻译官

### 25.3 情感与叙事体验

- 从甜蜜到失控
- 从恐怖回到现实
- 最终落到亲密关系中的真实问题

### 25.4 传播潜力

- 最大嘴硬句
- 反义污染次数
- 被读取草稿次数
- 读档失败次数
- 关系幻觉类型
- 一句话 AI 翻译

《过拟合恋人》的最终要求是：

- 以前半段 Meta 恐怖体验建立情绪张力
- 以后半段恋爱翻译官回收主题表达
- 用正式能力完成完整链路
- 不接受任何实质功能上的代偿交付
