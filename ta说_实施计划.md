# 《过拟合恋人》实施计划

## 1. 目标

本计划以 `ta说_PRD_最终版.md` 为准，目标是完成《过拟合恋人》的正式开发版本，而不是演示性质原型。

核心要求：

1. 以正式功能完成完整产品链路。
2. 不以 `MVP`、`Demo`、`mock` 或静态占位作为交付标准。
3. 不采用任何实质功能上的代偿实现。

## 2. 工作主线

### 开发者 A

负责：

- 首页 / 选择页 / 聊天主界面
- 反义污染
- 恐怖回复系统
- 输入框监听
- 读档系统
- 空间异常
- 真相页 / 梦醒页

### 开发者 B

负责：

- 恋爱翻译官
- 分享卡
- 语音输入
- 翻译相关接口与数据组装

## 3. 当前正式任务

1. 补齐聊天主链路的正式回复能力。
2. 补齐翻译官的正式结构化生成能力。
3. 补齐分享卡的正式数据链路与导出能力。
4. 补齐语音输入的真实能力接入。
5. 让文档、类型、接口和实现保持一致。

## 4. 协作规则

1. 共享类型变更必须先同步。
2. 文档要求变更必须先改 PRD，再同步其它文档。
3. 页面组件不得直接拼接模型请求。
4. 不允许以代偿实现标记功能“已完成”。

## 5. 风险与应对

### 风险一：模型输出不稳定

应对：

- 强约束 prompt
- 固定结构输出
- 明确错误提示与重试路径

### 风险二：实现用假数据顶替正式功能

应对：

- 文档明确禁止
- 代码评审按正式链路验收
- 关键能力必须基于真实状态与真实生成结果

### 风险三：文档和代码再次脱节

应对：

- PRD 作为唯一基准
- 共享契约变更后同步更新 README 与说明文档
