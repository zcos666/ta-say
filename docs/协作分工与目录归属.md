# 《ta说》协作分工与目录归属

## 1. 总原则

本项目采用“双主线并行 + 共享底座锁定”的协作方式。

- 开发者 A 负责前 80% 的恋爱 Meta 恐怖游戏体验。
- 开发者 B 负责后 20% 的恋爱翻译官、分享卡与通关后闭环。
- 共享目录由双方共同维护，但必须先同步再改。

## 2. 开发者 A 负责清单

页面：

- `src/pages/StartPage`
- `src/pages/SelectionPage`
- `src/pages/ChatPage`
- `src/pages/SpacePage`
- `src/pages/TruthPage`
- `src/pages/WakePage`

能力：

- `src/features/pollution`
- `src/features/reply`
- `src/features/drafts`
- `src/features/save-load`

交付目标：

1. 从首页到梦醒页完整可通关。
2. 第一次污染稳定出现。
3. 草稿监听和读档三状态成立。
4. 至少一个 Meta 恐怖支线可用。

## 3. 开发者 B 负责清单

页面：

- `src/pages/TranslatorPage`
- `src/pages/ShareCardPage`

能力：

- `src/features/translator`
- `src/features/share-card`
- `src/services/voice`

交付目标：

1. 翻译官可完成输入到结构化输出。
2. 分享卡可展示并可导出。
3. 通关后首页可直接进入翻译官。
4. 语音输入具备可用或降级方案。

## 4. 共享底座

以下目录为共享底座：

- `src/types`
- `src/config`
- `src/features/story`
- `src/services/api`
- `src/services/storage`
- `src/app/routes`
- `src/app/store`

共享底座职责：

- 保存全局状态
- 定义核心类型
- 统一 API 封装
- 统一存储 schema
- 管理页面路由

## 5. 禁止交叉越界

开发者 A 不应直接改：

- 翻译官输出 schema 的业务含义
- 分享卡字段设计
- 分享卡页面结构

开发者 B 不应直接改：

- 聊天状态机主逻辑
- 污染规则优先级
- 读档机制的保留字段

## 6. 必须同步的变更

以下变更必须先同步：

1. `SessionState` 增删字段
2. `PersistedState` 结构变更
3. `LoveTranslationReport` 结构变更
4. `ShareCardData` 结构变更
5. 路由路径变更
6. localStorage key 变更

## 7. 每次集成前检查

1. A 是否已经提供会话统计字段。
2. B 是否已经提供分享卡所需数据模型。
3. 首页是否正确读取 `hasFinishedGame`。
4. 共享类型是否一致。
5. 是否有人在共享目录留下临时 hack。

## 8. 推荐协作节奏

- 上午：各自推进主模块
- 下午：共享接口同步一次
- 晚上：合并并跑完整主链路

## 9. 最终目标

不是两个人分别做两个项目，而是做同一个状态机驱动的产品两段体验。本文件的作用就是防止后续开发重新裂成两个孤立系统。
