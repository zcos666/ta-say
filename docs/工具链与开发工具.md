# 《ta说》工具链与开发工具

## 1. 目标

本文件用于统一项目开发工具、推荐库和使用边界，避免两位开发者各自引入不同方案导致联调成本上升。

## 2. 前端工具链

推荐统一采用：

- `Vite`
- `React`
- `TypeScript`
- `Zustand`
- `React Router`
- `Tailwind CSS` 或 `CSS Modules`

优先级说明：

1. 能不用重型库就不用。
2. 能复用浏览器原生能力就不要额外包一层复杂封装。
3. 不为“看起来更工程化”而增加黑客松负担。

## 3. 开发工具

统一建议：

- IDE：`Trae`
- 包管理：`pnpm` 优先，若未统一则临时用 `npm`
- 代码格式：`Prettier`
- 代码检查：`ESLint`
- 单测：`Vitest`
- 截图生成：`html-to-image`

## 4. AI 与协作工具

### 4.1 Trae

用途：

- 快速搭页面骨架
- 生成局部模块
- 修明确 bug
- 补测试或 mock

限制：

- 不直接让它决定整体架构
- 不直接让它重构整个项目
- 不在无边界的情况下让它“自由发挥”

### 4.2 LLM API

用途：

- 反义污染补充生成
- TA 回复补充生成
- 恋爱翻译官结构化输出
- 分享卡一句话生成

边界：

- 剧情推进不交给 LLM
- 关键机制必须有本地 fallback

## 5. 浏览器能力

推荐原生使用：

- `localStorage`：存档、通关状态、最近一次分享卡
- `Web Speech API`：语音输入
- `Canvas` / `html-to-image`：分享卡导出

## 6. 测试工具

推荐：

- 单元测试：`Vitest`
- 组件测试：按需使用 `Testing Library`
- 手动联调：浏览器本地运行 + 真机尺寸模拟

- `src/features/story`
- `src/config`

## 8. 禁止随意新增的工具

以下内容在双方未确认前不要自行引入：

- Redux 或其他重型状态库
- 多套 UI 组件库
- 新的 CSS-in-JS 方案
- 新的测试框架
- 新的路由方案
- 新的后端运行时

## 9. 工具选型原则

当出现多个可选工具时，按以下顺序决策：

1. 是否能最快完成 MVP
2. 是否能降低联调成本
3. 是否能稳定跑 Demo
4. 是否方便做离线 fallback
5. 是否真的比现有方案更简单

## 10. Trae 提示词存放约定

- 共享提示词：`.trae/prompts/shared`
- A 的提示词：`.trae/prompts/developer-a`
- B 的提示词：`.trae/prompts/developer-b`

每个提示词文件建议包含：

- 目标
- 范围
- 限制
- 交付
- 验收
