# 《ta说》版本规范

## 1. 目标

本文件用于统一项目阶段、提交版本、演示版本和对外说明口径，避免出现“代码一版、文档一版、现场 Demo 又一版”的情况。

## 2. 当前版本定义

- 仓库基础版本：`v0.1.0-foundation`
- 当前阶段名称：`MVP Foundation`
- 含义：目录骨架、分工文档、架构文档、协作规则已建立，但功能代码尚未正式接入。

## 3. 版本分层

### 3.1 仓库版本

用于表示整个项目代码与文档状态。

格式：

```text
v主版本.次版本.修订号-阶段标签
```

示例：

- `v0.1.0-foundation`
- `v0.2.0-playable`
- `v0.3.0-demo`
- `v1.0.0-hackathon`

### 3.2 演示版本

用于现场或评审沟通。

示例：

- `Demo Alpha`：能跑主链路
- `Demo Beta`：已接主要交互
- `Demo Release`：用于评审展示

### 3.3 文档版本

核心文档头部可加：

- `文档状态：草案`
- `文档状态：可执行`
- `文档状态：已锁定`

## 4. 推荐里程碑版本

| 版本 | 阶段 | 含义 |
|---|---|---|
| `v0.1.0-foundation` | 当前 | 文档、目录、协作规则就绪 |
| `v0.2.0-playable` | 主链路可玩 | 首页到梦醒页打通，翻译官 mock 可用 |
| `v0.3.0-demo` | Demo 联调 | 读档、草稿监听、分享卡、A/B 集成完成 |
| `v0.4.0-online` | 在线版 | 接入真实 LLM 与语音 |
| `v1.0.0-hackathon` | 提交版 | 可演示、可截图、可离线降级 |

## 5. 版本升级规则

### 升次版本

满足任一条件时升级次版本：

- 主链路新增一段可演示体验
- A / B 两侧完成一次有效集成
- 核心功能从 mock 切到真实实现

### 升修订号

满足任一条件时升级修订号：

- 修复 bug
- 调整文案
- 优化样式
- 完善工具脚本或文档

## 6. 推荐提交流程

1. 功能完成后先本地自测。
2. 更新对应文档或 README。
3. 如果阶段有明显推进，更新 `CHANGELOG.md`。
4. 需要对外演示时，再更新演示版本说明。

## 7. CHANGELOG 书写规则

每条变更建议写清：

- `Added`：新增了什么
- `Changed`：改了什么
- `Fixed`：修了什么
- `Docs`：补了什么文档

## 8. A / B 的版本职责

开发者 A 重点更新：

- 主剧情交互变更
- 聊天机制变更
- 状态机变更
- 读档机制变更

开发者 B 重点更新：

- 翻译官输出结构变更
- 分享卡字段变更
- 导出能力变更
- 工具链和 API 接入变更

## 9. 版本锁定点

以下节点建议锁版本并通知对方：

- 共享类型改动
- 路由改动
- localStorage schema 改动
- `ShareCardData` 改动
- `LoveTranslationReport` 改动
